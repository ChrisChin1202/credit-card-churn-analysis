# 信用卡客戶流失分析

---

## 為何製作這個專案

我想用一個「做得完整、也講得清楚」的專案，證明我真的能把資料變成有用的判斷。
我會挑「信用卡客戶流失」這個題目，是因為它是銀行、電商、訂閱制服務每天都在煩惱的事——留住一個老客戶，通常比拉一個新客戶便宜很多。如果能在客戶真的離開「之前」就認出「誰快走了」，公司就能把有限的預算花在對的人身上。
在專案當中，我把自己想像成一位銀行信用卡部門的初階分析師，公司給我三個問題：

1. 我們現在的流失率有多高？
2. 會走的客戶，長什麼樣子？
3. 預算有限，要先留住誰？

---

## 先講結論

**客戶通常不是突然消失的，而是先變得不活躍、交易越來越少，然後才流失。** 所以只要盯著「交易次數在下滑」的客戶提早關心，就有機會把人留下來。

- 流失客戶平均一年只刷 **45 次**卡，留下來的人刷了 **69 次**。
- **跟銀行往來越深，越不會走**：只辦 1～2 項產品的人流失率將近 **28%**，辦了 4 項以上的只剩約 **11%**。
- **白金卡流失率最高（25%）**，反而是高卡等的客戶比較容易被挖走。
- 我寫了一個模型，把每位客戶標成高／中／低風險。事後驗證，被標成「**高風險**」的人實際流失率 **51.4%**，「低風險」只有 **1.4%**——差了快 **37 倍**，代表這份名單真的找得出該優先關心的人。

## 我用了哪些工具，各做了什麼

我讓每個工具做它最擅長的事，貼近真實工作流程。

### SQL（SQLite）— 資料清洗

我在 SQLite 裡把資料匯入、做品質檢查:

- 原始資料裡有兩個欄位，是「別人事先算好的預測結果」。如果留著，模型等於資料洩漏，所以我把它們刪掉。
- 還有一欄（`Avg_Open_To_Buy`）其實是另外兩欄相減算出來的，跟其中一欄幾乎一模一樣（**共線性**），留著會讓後面模型的判斷亂掉，我也移除了。


### Python — 、建立模型比較資料

我用 pandas 整理資料、畫圖比較「流失」跟「留存」兩群人的差異。

我用**邏輯迴歸**建了一個預測流失的模型。它解釋「為什麼會流失」模型抓得出大約 8 成真的會流失的人（**recall 0.82**），整體區分能力也不錯（**AUC 0.92**）。最後我用模型替每一位客戶算出流失機率、分成高中低風險，輸出成一份排序好的名單。

### 📊 Excel — 交叉分析

我用 Excel 樞紐分析表，把各客群的流失率交叉比一比，再用條件式格式把高風險客群標成紅色、做成一張熱力圖。

### 📈 Tableau — 互動式洞察儀表板

我把客群洞察做成可以點選互動的 Tableau 儀表板，發佈到 Tableau Public，任何人都能點開自己探索——點某個卡別，三張圖會一起連動。

🔗 **線上互動版**：https://public.tableau.com/app/profile/chinyuchn/viz/_17816910160820/1_1

### 🧭 Power BI — 高風險預警名單

最後我做了一份 Power BI 儀表板，定位跟 Tableau 不一樣：它是給第一線業務「今天該打給誰」用的營運工具。上面有 KPI 卡片、一張證明模型分級有效的長條圖，還有一份依流失機率由高到低排序的高風險客戶清單，業務從最上面開始聯繫就對了。

## 小插曲

做 Excel 的時候，我一度看到「不活躍 0 個月」的客群流失率高達 **51.7%**，差點興奮地把它寫進結論。還好我多看了一眼樣本數——這群其實只有 **29 個人**，根本是雜訊。後來做交叉表又遇到一個「100% 流失」的紅格，一查發現那格只有 **1 個人**。

這件事讓我學到：**看比率一定要同時看樣本數，不然很容易因為樣本數不足而沒有結論。** 
---

## 專案的限制

- 用的是 Kaggle 的公開練習資料，不是真實銀行資料，所以結論不能直接套用到任何一家公司。
- 只有一個時間點的橫斷面資料，沒有每月趨勢，真實情境最好接上時間序列。
- 收入跟流失的關係其實不太明顯（高收入和低收入兩端都偏高、中間低，但各級距差距很小），沒有結論因為資料還不足以下定論。
- 如果有後續追蹤我會想再追蹤：被標成高風險的人後來真的成為流失客戶了嗎？提早介入到底有沒有用？這些都得靠時間和真實資料才能驗證。
